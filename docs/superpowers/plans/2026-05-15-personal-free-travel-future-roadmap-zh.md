# 个人自由行后续功能路线文档

> **定位：** 本文档记录 V1 功能闭环之后，RealTrip Assistant 面向个人自由行用户继续演进时应建设的功能。它不是单次实施计划，而是后续功能池和阶段路线。每个阶段开始前，应再拆成独立 implementation plan。

**产品方向：** 从“一次性行程生成器”升级为“个人自由行全程助手”：出发前帮用户做选择，出发中帮用户应对变化，出发后沉淀偏好和复用经验。

**适用用户：** 个人自由行用户，包括独自出行、情侣/朋友出行、亲子游、带老人出行、轻度定制需求用户。

**不做方向：** 不做交易闭环，不代用户支付，不代用户预订，不做旅行社 ERP，不做企业后台套件。

---

## 总体阶段

| 阶段 | 名称 | 核心目标 | 优先级 |
|---|---|---|---|
| V1.5 | 体验补强 | 让生成后的行程更好改、更好看、更可信 | 高 |
| V2 | 个性化记忆 | 让系统越来越懂用户和同行人 | 高 |
| V2.5 | 出行执行助手 | 让用户能拿着行程真正出门 | 中高 |
| V3 | 途中动态助手 | 天气、体力、排队、临时变更时能实时调整 | 中高 |
| V3.5 | 内容和协作 | 分享、多人协同、旅行笔记、复用历史经验 | 中 |
| V4 | 高级智能规划 | 更强的路线优化、预算优化、偏好学习和质量评估 | 中 |

## V1.5：体验补强

### 目标

V1 已经能给出多方案、解释、清单和初步局部修改。V1.5 的目标是把这些功能打磨成用户“看得懂、改得动、信得过”的体验。

### 功能 1：方案对比增强

当前方案对比主要是预算、活动数、天数等基础维度。后续应扩展为更像个人用户的决策面板。

应增加：
- 每个方案的一句话结论。
- 适合人群：第一次去、亲子、带老人、情侣、预算有限。
- 风险提示：步行多、换乘多、热门景点拥挤、天气依赖强。
- 预算构成：交通、住宿、餐饮、门票、其他。
- 体力强度：低、中、高。
- 自由时间比例：每天是否留有空档。
- 方案推荐排序：系统默认推荐一个最均衡方案。

验收标准：
- 用户不用展开完整行程，也能判断该选哪个方案。
- 至少能区分“省钱但体验弱”“经典但较累”“轻松但覆盖少”。

### 功能 2：行程差异展示

用户修改行程后，需要清楚知道系统改了什么。

应增加：
- 修改前后 diff。
- 被删除的景点。
- 新增的景点。
- 时间变化。
- 预算变化。
- 为什么要这样改。
- 是否影响已锁定项目。

典型场景：
- “第二天太累了，轻松一点。”
- “把博物馆换成美食街。”
- “预算降到 3000。”
- “下雨就换室内。”

验收标准：
- 每次修改后都能生成结构化变更摘要。
- 用户能一键接受或撤销本次修改。

### 功能 3：解释能力升级

V1 的解释卡是规则化解释。后续应变成更细的“推荐理由 + 风险 + 替代项”。

应增加：
- 推荐理由：匹配用户哪些偏好。
- 排程理由：为什么放在上午/下午/晚上。
- 地理理由：是否顺路、是否靠近酒店或下一个点。
- 风险：排队、闭馆、天气、交通、体力。
- 替代项：同类型可替换点。
- 信息来源：实时查询、模型知识、用户提供。

验收标准：
- 用户点击任意 segment，都能看到至少三类解释。
- 动态信息必须标注查询时间或来源类型。

### 功能 4：快捷修改入口

不要只依赖用户自己输入自然语言。个人用户更喜欢点选式修改。

应增加：
- “这天太累”
- “换成室内”
- “少花钱”
- “多安排美食”
- “别太早起”
- “保留这个”
- “换一个类似的”
- “删掉这个”

验收标准：
- 每个 segment 和 day 都有上下文相关的快捷操作。
- 快捷操作最终仍转成统一 revision intent，避免前后端各自写逻辑。

## V2：个性化记忆

### 目标

让系统从“每次重新认识用户”变成“长期知道用户怎么旅行”。

### 功能 1：用户旅行偏好档案

应记录：
- 默认出发城市。
- 常用预算区间。
- 喜欢的旅行节奏。
- 每天可接受最早出发时间。
- 每天最多主要景点数量。
- 餐饮偏好：本地小吃、清淡、素食、儿童友好、不吃辣。
- 酒店偏好：地铁附近、安静、亲子、价格上限。
- 交通偏好：高铁、飞机、自驾、少换乘。
- 避免项：网红打卡、购物、太多博物馆、早起、夜间活动。

验收标准：
- 用户第二次规划时，系统能自动带入偏好。
- 用户能查看、修改、删除偏好。
- 系统应区分“用户明确说过”和“系统推断”。

### 功能 2：同行人画像

个人自由行经常不是一个人。同行人会显著影响行程。

应支持：
- 成人、儿童、老人。
- 年龄段。
- 体力水平。
- 行动不便。
- 饮食禁忌。
- 午休需求。
- 是否适合长时间步行。
- 是否需要亲子设施。

典型画像：
- “带 2 岁孩子。”
- “带 70 岁老人。”
- “情侣第一次去。”
- “朋友四人，想吃好玩好。”

验收标准：
- 带老人时，行程自动降低强度、增加休息、减少换乘。
- 带儿童时，增加亲子友好和午休/早睡考虑。

### 功能 3：偏好学习

系统应从用户修改行为中学习。

可学习信号：
- 用户频繁删除博物馆。
- 用户多次要求不要早起。
- 用户总是保留美食安排。
- 用户多次选择省钱方案。
- 用户经常把户外换成室内。

需要注意：
- 学习结果必须可解释。
- 高影响偏好需要用户确认。
- 用户可以关闭记忆。

验收标准：
- 系统能生成“我记住了：你不喜欢太早出发”这类确认。
- 偏好不会静默污染后续行程。

## V2.5：出行执行助手

### 目标

让用户不只是看方案，而是真的能按行程出发。

### 功能 1：出发前清单持久化

V1 的清单可以先临时生成。后续应持久化并支持勾选。

应支持：
- 每个 trip 独立清单。
- 用户勾选状态保存。
- 自动生成和手动添加并存。
- 分类：证件、交通、住宿、门票、天气、装备、预算。
- 优先级：必须、建议、可选。

验收标准：
- 刷新页面后勾选状态不丢失。
- 修改行程后，清单能增量更新。

### 功能 2：预算明细

个人用户很关心钱，但不一定需要复杂财务系统。

应支持：
- 总预算。
- 预计总花费。
- 按天花费。
- 按类别花费：交通、住宿、餐饮、门票、其他。
- 超预算提醒。
- 降预算建议。

验收标准：
- 用户能看到“为什么超预算”。
- 系统能建议“减少一个高价项目/换住宿区域/减少打车”。

### 功能 3：预约和营业时间提醒

应支持：
- 景点是否需要预约。
- 门票是否建议提前购买。
- 闭馆日提醒。
- 最晚入园时间。
- 节假日拥挤提示。

验收标准：
- 行程中有预约风险时，在清单和 segment 详情中同时提示。

### 功能 4：导出与分享升级

应支持：
- PDF。
- Word。
- Markdown。
- 日历 ICS。
- 简洁分享链接。
- 同行人阅读版。
- 老人友好版：字体更大、信息更少、重点突出。

验收标准：
- 用户可以把行程发给同行人。
- 导出内容不是数据库字段堆叠，而是可读的旅行文档。

## V3：途中动态助手

### 目标

自由行最难的是计划赶不上变化。V3 要让系统能在途中帮用户快速调整。

### 功能 1：天气驱动重排

应支持：
- 雨天自动替换户外项目。
- 高温减少暴晒和步行。
- 寒冷天气减少户外停留。
- 台风/暴雨等极端天气风险提示。

验收标准：
- 用户输入“今天下雨怎么办”，系统只调整受影响的当天或半天。
- 调整后保留酒店、交通、已锁定项目。

### 功能 2：体力和时间驱动重排

应支持：
- “我现在太累了。”
- “今天起晚了。”
- “这个景点排队太久。”
- “我们想早点回酒店。”
- “孩子睡着了，下午轻松点。”

验收标准：
- 系统能裁剪当天剩余安排。
- 系统能把低优先级项目移到备选，而不是直接消失。

### 功能 3：实时路线辅助

应支持：
- 当前点到下一个点的交通建议。
- 步行/地铁/打车对比。
- 是否来得及赶到下一站。
- 如果来不及，建议跳过或替换。

验收标准：
- 每次路线建议都说明预计时间和缓冲时间。

### 功能 4：半日重排

不应该每次重排整趟旅行。

应支持：
- 只重排上午。
- 只重排下午。
- 只重排今晚。
- 只重排当前点之后。

验收标准：
- 已完成和已锁定的安排不会被改。
- 用户能看到“未动哪些内容”。

## V3.5：内容和协作

### 目标

让行程不只是个人私有文档，而是能分享、讨论、复用和沉淀。

### 功能 1：同行人协作

应支持：
- 分享只读行程。
- 同行人点赞/评论某个景点。
- 同行人提出“想去/不想去”。
- 主用户统一接受或拒绝。

验收标准：
- 同行人不能直接破坏主行程。
- 所有协作修改都进入待确认队列。

### 功能 2：旅行笔记

应支持：
- 每天记录实际体验。
- 标记实际花费。
- 标记某个推荐是否满意。
- 上传文字备注。

验收标准：
- 出行后能生成“实际旅行记录”。
- 用户反馈能反哺偏好学习。

### 功能 3：历史行程复用

应支持：
- 复制一份历史行程。
- 换日期重排。
- 换同行人重排。
- 换预算重排。
- 复用某一天作为模板。

验收标准：
- 用户能基于“去年杭州 3 日游”生成“今年带父母杭州 4 日游”。

## V4：高级智能规划

### 目标

让系统从“会生成和修改”进一步变成“会优化和评估”。

### 功能 1：路线优化

应支持：
- 减少往返路。
- 减少跨城/跨区跳跃。
- 酒店区域和景点组合联动。
- 根据交通时间重排当天顺序。

验收标准：
- 每天路线有清晰地理逻辑。
- 系统能解释“为什么先去 A 再去 B”。

### 功能 2：预算优化

应支持：
- 在不牺牲核心体验的情况下降低预算。
- 给出三种降预算策略。
- 标出最影响体验的高价项。

策略示例：
- 换住宿区域。
- 减少打车。
- 替换高价门票。
- 减少跨城移动。

验收标准：
- 用户输入“预算从 8000 降到 6000”，系统能给出可执行方案和牺牲点。

### 功能 3：行程质量评分

应支持：
- 预算合理性评分。
- 体力强度评分。
- 路线顺畅度评分。
- 天气风险评分。
- 亲子/老人友好评分。
- 信息可信度评分。

验收标准：
- 每个方案都有质量评分。
- 评分必须能展开解释，不允许只有一个数字。

### 功能 4：评测集和案例库

为了持续提升功能质量，需要建设典型个人自由行案例。

案例类型：
- 第一次去北京 5 天。
- 成都美食 4 天。
- 带老人去杭州 3 天。
- 带儿童去上海迪士尼 3 天。
- 云南轻松 7 天。
- 预算有限的广州深圳 4 天。
- 雨天备选明显的城市游。

验收标准：
- 每次核心规划逻辑修改后，能跑案例集对比质量。
- 输出不仅看是否成功，还要看预算、节奏、风险和可解释性。

## 功能优先级建议

### 最高优先级

1. 行程差异展示。
2. 快捷修改入口。
3. 用户偏好档案。
4. 同行人画像。
5. 出发前清单持久化。

这些功能直接决定产品是否像一个真正懂自由行的助手。

### 中高优先级

1. 预算明细。
2. 预约提醒。
3. 天气驱动重排。
4. 半日重排。
5. 导出分享升级。

这些功能会明显提高真实出行可用性。

### 中优先级

1. 同行人协作。
2. 旅行笔记。
3. 历史行程复用。
4. 路线优化。
5. 质量评分。

这些功能适合作为产品进入稳定期后的增强能力。

## 推荐实施顺序

### 第一轮：V1.5 体验补强

先做：
- 行程差异展示。
- 快捷修改入口。
- 解释能力升级。

原因：
- 当前代码已经有 `PlanDiff`、revision、pins、plan version 的基础。
- 做完后，用户会明显感觉“生成后还能继续聊、继续改”。

### 第二轮：V2 个性化记忆

再做：
- 用户旅行偏好档案。
- 同行人画像。
- 偏好学习确认机制。

原因：
- 个人自由行产品最重要的壁垒是“越来越懂我”。
- 这部分会影响后续所有生成和修改质量。

### 第三轮：V2.5 出行执行助手

再做：
- 清单持久化。
- 预算明细。
- 预约提醒。
- 导出分享升级。

原因：
- 用户会从“看行程”进入“照着行程走”。

### 第四轮：V3 途中动态助手

最后做：
- 天气重排。
- 体力/时间重排。
- 实时路线辅助。
- 半日重排。

原因：
- 这部分非常有价值，但依赖前面已有稳定的局部修改、锁定、差异展示和偏好系统。

---

## 需要避免的方向

1. 不要过早做复杂后台。
2. 不要把功能做成旅行社工作台。
3. 不要把所有查询都交给 LLM 判断。
4. 不要每次修改都重排整趟行程。
5. 不要在用户没确认时静默学习偏好。
6. 不要做预订和支付闭环。
7. 不要用太多“看起来高级但用户不懂”的评分。
