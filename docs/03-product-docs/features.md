## Phase 1：基础规划

### 1.1 CLI 交互式需求收集

通过命令行对话逐项收集旅行约束条件：

```
目的地：杭州
出发日期：2026-05-01
天数：4
人数：4人（含2位老人）
总预算：20000 CNY
节奏偏好：慢
兴趣方向：文化, 自然
```

支持中途退出（Ctrl+C）和输入确认环节。

### 1.2 行程规划生成

根据约束条件，调用 LLM 引擎生成结构化行程，包含：

- **每日时间线**：以 30 分钟为单位的活动编排
- **活动类型标注**：文化/自然/美食/购物/交通/住宿
- **费用估算**：每项活动预估花费
- **位置信息**：活动所在地点

### 1.3 基础规则校验

每次生成的行程自动通过 4 条确定性规则校验：

| 规则 ID | 名称 | 说明 |
|---------|------|------|
| R01 | 时间连续性 | 同一天相邻活动时间不得重叠 |
| R02 | 预算 | 所有活动费用之和 ≤ 预算上限 |
| R03 | 日期边界 | 所有行程日期在旅行范围内 |
| R04 | 必填完整性 | 每天至少包含一个主活动 |

校验失败自动触发修订流程，最多重试 3 轮。

### 1.4 Markdown 行程输出

最终行程输出为结构化 Markdown 文件，包含：

- YAML 前头（目的地、日期、人数、预算、状态）
- 每日时间线（含时间、活动名、类型标签）
- 每项活动的证据来源（URL 或模型知识标注）

### 1.5 文件持久化

所有数据保存到文件系统：

- `trip.md`：可读可编辑的 Markdown 行程
- `state.json`：结构化状态数据
- `evidence/`：独立的证据文件

## Phase 2：高级规划能力

### 2.1 4 Agent 协同架构

系统拆分为 4 个专职 Agent 协作：

| Agent | 职责 | 用户可见 |
|-------|------|---------|
| **Supervisor** | 调度中枢，状态机管理，异常降级 | 否 |
| **Researcher** | 收集 POI、天气、交通等信息 | 是（来源标签） |
| **Planner** | 编排行程，生成候选方案 | 是（生成结果） |
| **Verifier** | 规则校验 + 语义判断 + 风险检查 | 是（校验报告） |

### 2.2 完整规则校验体系（8 + 3）

在 Phase 1 的 4 条规则基础上增加：

**确定性规则（代码）：**

| 规则 ID | 名称 | 说明 |
|---------|------|------|
| R05 | 空间连续性 | 活动间留有充足交通时间（buffer ≥ ETA × 1.3） |
| R06 | 开放时间 | 活动时间在景点开放时间内（需证据） |
| R07 | 行程密度 | 每日主活动不超过阈值（慢 ≤ 2，中 ≤ 4，快 ≤ 6） |
| R08 | 用户锁定项 | pinned 项目未被修改 |

**语义判断（LLM，仅在确定性规则全部通过后触发）：**

| 规则 ID | 名称 | 说明 |
|---------|------|------|
| S01 | 节奏合理性 | 整体节奏是否匹配用户偏好 |
| S02 | 多样性 | 多日活动类型是否过于单一 |
| S03 | 逻辑连贯 | 同日活动动线是否合理 |

### 2.3 用户锁定（User Pinning）

用户可锁定以下类型安排：

- **景点**：锁定后不可被替换或删除
- **酒店**：锁定后重规划不可变更
- **餐厅**：锁定后保持预订信息
- **日期/预算**：锁定后作为硬约束

锁定范围支持"整个行程"或"指定某天"。Verifier 的 R08 自动检查 pin 完整性。

### 2.4 方案差异对比（Plan Diff）

每次重规划后自动输出结构化变化：

```
本次调整：
1. Day 2 上午"西湖游船"→"浙江省博物馆"
   原因：天气预报中雨
   影响：预算减少 120 元，步行减少 1.2km
2. Day 3 酒店未变（用户已锁定）
```

支持查看任意两个历史版本间的差异。

### 2.5 假设管理（Assumption Ledger）

系统将默认设定分为两级管理：

- **隐式假设**：不主动打扰用户，可在"假设清单"中查看修改（如默认午餐时间 12:00、默认酒店入住 14:00）
- **显式假设**：必须用户确认（如"老人每日步行上限 6000 步"、"预算是否含餐饮"）

最多同时展示 3 条待确认假设，避免信息过载。

### 2.6 分级降级策略

当外部服务不可用时，系统按四级策略降级：

| 层级 | 策略 | 用户体验 |
|------|------|---------|
| L1 | 自动重试 1 次（间隔 2s） | 无感 |
| L2 | 用 LLM 知识兜底，标记来源 | 可见来源标签 |
| L3 | 暂停流程，向用户提问 | 等待输入 |
| L4 | 标记当前阶段失败，保留已有成果 | 输出部分结果 + 失败说明 |

### 2.7 SQLite 缓存层

- `cache.db`：工具调用结果缓存（自动过期），减少重复调用
- `cost_log.db`：LLM 调用成本记录，为 Phase 3 做预算评估

## Phase 3：产品化功能

### 3.1 Web UI 可视化

- **行程时间线**：可视化每日活动卡片，拖拽调整顺序
- **地图视图**：行程点在地图上标注，直观查看动线
- **校验状态**：每条规则通过/失败一目了然

### 3.2 多会话管理

- 同时规划多个旅行（如杭州和三亚），互不干扰
- 历史行程归档和检索
- 会话状态自动保存，关闭页面后恢复

### 3.3 方案对比

- 并行生成 2-3 个候选方案（经济型/舒适型/深度型）
- 并排对比：预算、活动数、节奏、步行距离等多维度对
- 差异高亮标记，快速定位不同点
- 共享 Researcher 结果，降低成本

### 3.4 用户系统

- 注册/登录
- 多用户数据隔离
- 个人偏好管理

### 3.5 长期偏好学习

系统跨行程学习用户偏好，且区分上下文：

- **带老人出游** → 慢节奏（高置信度）
- **独自城市游** → 快节奏（中置信度）
- **家庭出游** → 中等预算（高置信度）

偏好需用户授权保存，随时可删除。

### 3.6 导出中心

| 格式 | 适用场景 |
|------|---------|
| **Markdown** | 自由编辑、分享 |
| **PDF** | 打印、离线查看 |
| **日历 ICS** | 导入手机日历 |

### 3.7 目的地知识库

手工维护的结构化知识库（Markdown 格式）：

- 景点开放时间、门票价格
- 推荐游玩时长
- 季节/天气适配建议
- 特殊注意事项（如周一闭馆）

用户规划时系统通过 RAG 检索相关知识作为证据。

### 3.8 Reflection Agent（异步）

行程完成后异步执行，不阻塞主流程：

- 分析行程特征，提取偏好信号
- 分析用户修改项（如手动调整时间 → 偏好线索）
- 将高置信度偏好推送给用户确认

## Phase 4：企业级功能

### 4.1 全链路可观测

- **Trace Recorder**：记录每次 Agent 调用、工具调用、LLM 调用的完整链路
- **Replay Engine**：回放历史会话，复现故障
- **OpenTelemetry**：标准追踪接入
- **Grafana Dashboard**：延迟、成本、成功率实时监控

### 4.2 模型路由（Model Router）

按任务复杂度智能路由到不同模型：

| 任务类型 | 模型 | 理由 |
|---------|------|------|
| 意图识别 | Haiku | 简单，低延迟低成本 |
| POI 信息提取 | Sonnet | 中等复杂度 |
| 行程规划 | Opus | 需要高精度 |
| 语义校验 | Sonnet | 中等复杂度 |
| 偏好提取 | Haiku | 简单 |

### 4.3 知识图谱

结构化目的地关系图：

- 景点相邻关系（西湖 → 灵隐寺，8km）
- 交通方式可达性（杭州 → 乌镇，大巴 1.5h）
- 季节适配（5 月杭州 → 适宜，7 月杭州 → 炎热）
- 推荐组合（灵隐寺 + 浙江省博物馆 = 一日文化游）

支持语义查询："西湖附近有什么适合带老人去的景点？"

### 4.4 A/B Diff + Golden Set CI

- Golden Set：10-20 个典型场景测试用例
- CI 集成：每次变更自动跑回归测试
- A/B Diff：Prompt/模型变更时自动对比新旧方案质量

### 4.5 全链路成本追踪

| 指标 | 说明 |
|------|------|
| cost_per_plan | 单次规划平均成本 |
| cost_per_comparison | 方案对比场景成本 |
| model_distribution | 各模型调用占比 |
